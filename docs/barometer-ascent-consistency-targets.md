# 气压计累计爬升一致性目标说明

设计日期：2026-05-19

本文定义真实徒步过程中，多台设备在同一时间、同一条路线下使用气压计方案计算
累计爬升时的目标误差范围。这里的目标重点是**多设备一致性**，不是和某个地图、
DEM 或人工标注真值的绝对一致性。

## 1. 适用范围

适用场景：

- 多台设备同时出发或时间差很小，经历基本相同的天气气压变化。
- 设备走同一条真实徒步路线，路线方向一致，停留点和移动节奏接近。
- 主累计爬升来源为 `BAROMETER`，即 `Sensor.TYPE_PRESSURE` 压力样本换算出的
  相对高度变化。
- 对比的是同一次活动结束后的 `barometerTotalAscentMeters` 或
  `selectedTotalAscentMeters(source=BAROMETER)`。

不适用场景：

- 不同日期、不同天气窗口的路线复测。
- 一台设备使用气压计，另一台设备仅使用 GNSS 或 DEM。
- 设备经历不同交通工具、电梯、缆车、室内外压力突变。
- 设备佩戴位置明显不同，例如一台长期放在背包深处，另一台手持暴露在风口。

## 2. 目标口径

同一时间同一路线下，气压变化背景基本一致，因此多设备结果应该比跨天复测更集中。
但真实徒步中仍会受到佩戴位置、温度、气流、传感器批次、采样间隔、停走节奏和滤波
滞后的影响。第一版验收不追求过窄的实验室指标，目标范围适当放宽，优先保证真实
户外稳定可解释。

推荐目标：

| 设备组合 | 优秀目标 | 可接受目标 | 排查线 |
| --- | ---: | ---: | ---: |
| 同型号、同固件、同算法 | `<= 5%` | `<= 8%` | `> 12%` |
| 同品牌不同型号/传感器 | `<= 8%` | `<= 12%` | `> 15%` |
| 不同品牌、同算法策略 | `<= 8%` | `<= 12%` | `> 18%` |
| 不同品牌且算法不同 | `<= 10%` | `<= 15%` | `> 20%` |

“不同品牌、同算法策略”表示滤波、趋势确认、阈值和活动归因口径一致，但传感器型号、
机身结构、压力孔位置、系统采样策略可能不同，因此目标不应按同型号设备收紧。

产品验收建议以同型号设备为主：

```text
同型号多设备、同一时间、同一路线：
  目标：累计爬升设备间相对偏差 <= 8%
  红线：累计爬升设备间相对偏差 > 12% 时必须进入样本排查
  优秀：稳定 <= 5%
```

这里的“相对偏差”建议用设备集合中位数作为基准：

```text
relativeDeviation = abs(deviceAscent - medianAscent) / max(medianAscent, minReferenceAscent)
```

其中 `minReferenceAscent` 建议取 `100m`，避免低爬升路线中百分比被少量噪声放大。

## 3. 按路线爬升量分档

低爬升路线更适合看绝对差值，中高爬升路线再看百分比。第一版建议同时记录绝对差和
相对差。

| 路线累计爬升中位数 | 同型号多设备合理差异 | 说明 |
| ---: | ---: | --- |
| `< 50m` | `<= 15m` | 不建议用百分比验收，容易被传感器噪声放大。 |
| `50m - 100m` | `<= 20m` | 小坡、小台阶、短路线的绝对差优先。 |
| `100m - 300m` | `<= 25m` 或 `<= 12%` | 两者满足其一即可视为可接受。 |
| `300m - 800m` | `<= 8%`，红线 `12%` | 主要验收区间。 |
| `800m - 1500m` | `<= 8%`，优秀 `5%` | 长路线可期望百分比更稳定。 |
| `> 1500m` | `<= 6% - 8%` | 长距离若天气窗口一致，设备间离散应继续收敛。 |

示例：

```text
路线中位爬升 500m:
  优秀范围约 475m - 525m
  可接受范围约 460m - 540m
  若出现 < 440m 或 > 560m，需要排查

路线中位爬升 120m:
  可接受差异优先看 <= 25m
  不因单台偏差略超过 8% 就直接判失败
```

## 4. 验收样本要求

为了让目标能真实反映户外表现，建议至少覆盖以下样本：

| 样本类型 | 建议数量 | 目标 |
| --- | ---: | --- |
| 短低爬升路线 | `>= 3` | 验证小爬升不被气压抖动累计放大。 |
| 中等爬升路线 | `>= 5` | 作为主验收样本，覆盖普通徒步节奏。 |
| 长爬升路线 | `>= 2` | 验证长时间气压变化和采样稳定性。 |
| 频繁停走路线 | `>= 2` | 验证休息、拍照、补给时压力波动不会明显虚增。 |
| 弱 GNSS/林区路线 | `>= 2` | 验证 BAROMETER 不依赖 `moving_good_fix` 后仍可稳定累计。 |

每组样本建议至少 3 台同型号设备；如果条件允许，用 5 台设备更容易发现离群设备。

## 5. 统计方法

多设备自动验收必须有设备分组信息。每个 session 至少应携带：

```text
sessionId
strategyVersion
deviceManufacturer
deviceBrand
deviceModel
deviceName
androidSdkInt
selectedAscentSource
barometerTotalAscentMeters
barometerAscentSampleCount
barometerAscentRejectedSampleCount
```

其中设备字段由 App 自动写入 `session.json` 与 `session_metadata`。`routeId`、
`testBatchId`、`firmwareVersion`、`algorithmVersion` 可以先由验收 Web 导入时
补录；如果后续 App 支持测试批次配置，再直接写入导出数据。

单次路线统计：

```text
1. 过滤 source != BAROMETER 或样本明显异常的记录。
2. 计算所有设备累计爬升的 medianAscent。
3. 计算每台设备相对 medianAscent 的 absoluteDeviation 和 relativeDeviation。
4. 输出 maxDeviation、p80Deviation、离群设备列表。
```

推荐判定：

```text
通过:
  同型号设备 maxRelativeDeviation <= 8%
  或低爬升路线 absoluteDeviation 满足分档要求

观察:
  maxRelativeDeviation 在 8% - 12%
  且能解释为佩戴位置、气流、短时压力扰动或设备采样断层

失败:
  maxRelativeDeviation > 12%
  或多条路线同一设备持续偏高/偏低
```

不建议只用最大值和最小值直接相除，因为单台异常设备会把整组结果放大。验收报告应保留
中位数、最大偏差和离群设备，便于回看诊断日志。

## 6. 需要排查的典型信号

超过目标范围时优先排查：

- `barometerAscentSampleCount` 是否明显少于其他设备。
- `barometerAscentRejectedSampleCount` 是否异常偏高。
- 压力样本是否出现长时间断层或 elapsed time 倒退。
- 同一设备是否在多条路线持续偏高或偏低。
- 设备佩戴位置是否导致温度、气流或衣物覆盖差异。
- 是否存在交通工具、电梯、缆车、隧道、室内外快速切换。
- 滤波参数、`climbThreshold`、`dropThreshold` 是否导致短陡坡低估或小抖动虚增。

## 7. 与当前实现的关系

当前 `TrackAscentCalculator` 已经把 BAROMETER 与 GNSS 作为两条独立累计爬升链路，
并在有可靠气压计结果时优先选择 `BAROMETER`。本文目标不要求调整当前算法阈值，
而是作为真实徒步样本回放和多设备对比时的验收边界。

第一版结论：

```text
同一时间、同一路线、同型号多设备：
  累计爬升一致性目标放宽到 <= 8%
  超过 12% 进入排查
  小爬升路线按 15m - 25m 绝对差优先判断
```
